# Архитектура FinApp

## 1) Контекст

FinApp — мобильная платформа для управления личными финансами.
Архитектура построена как **микросервисный монорепозиторий** с единым мобильным клиентом и общей PostgreSQL.

## 2) Архитектурный стиль

- **Monorepo** для согласованной эволюции клиентского и серверного кода.
- **Микросервисы по доменам ответственности**.
- **ML как отдельный сервис** (FastAPI), вызываемый backend-сервисами.
- **Асинхронная обработка и кэширование** через Redis.

## 3) Верхнеуровневые компоненты

### Клиент
- `apps/mobile` (React Native + TypeScript): единое приложение iOS/Android.

### Backend (доменные сервисы)
- `auth` (Go + Gin): JWT, refresh tokens, базовая идентификация пользователя.
- `collection` (Go + Gin): импорт CSV/Excel, ручной ввод, приём голосовых данных.
- `processing` (Go + Gin): категоризация, обогащение транзакций, интеграция с ML.
- `subscription-detector` (Go + Gin): batch-анализ повторяющихся списаний.
- `analysis-control` (Java + Spring Boot): аналитика, бюджеты, цели, отчёты.

### ML
- `ml-service` (Python + FastAPI):
  - распознавание речи (Whisper Large-v3),
  - извлечение сущностей (NER RuBERT-tiny),
  - категоризация (CatBoost + CNN ансамбль).

### Data layer
- PostgreSQL: единая БД.
- Redis: кэш, промежуточные очереди, ускорение синхронизации.

## 4) Модульность по этапам ВКР

### Этап 1 — сбор и обработка данных
Покрывается сервисами `collection`, `processing`, `subscription-detector`, `ml-service`.
Ориентир: качественный ingestion-пайплайн и точная категоризация.

### Этап 2 — анализ и контроль
Покрывается сервисом `analysis-control`.
Ориентир: управленческая ценность для пользователя (бюджеты, цели, дашборд, отчёты).

## 5) Границы данных

### Таблицы зоны ответственности Модуля 1 (Go)
`users`, `categories`, `transactions`, `imports`, `subscriptions`, `voice_transcriptions`, `counterparties`, `sync_queue`.

### Таблицы зоны ответственности Модуля 2 (Java)
`budgets`, `goals`, `goal_transactions`, `reports`, `dashboard_widgets`.

### Общие таблицы
`notifications`, `notification_templates`, `recommendations`, `audit_logs`.

## 6) Потоки данных (основные сценарии)

### Сценарий A: Импорт выписки
1. Пользователь загружает CSV/Excel в мобильном приложении.
2. `collection` валидирует формат, нормализует данные и сохраняет импорт.
3. `processing` обогащает транзакции и запрашивает ML-категоризацию.
4. Результаты сохраняются в `transactions`; при необходимости формируются рекомендации.

### Сценарий B: Голосовой ввод транзакции
1. Клиент отправляет голосовой фрагмент в `collection`.
2. `collection` передаёт аудио в `ml-service` (ASR + NER).
3. Извлечённые поля транзакции подтверждаются/корректируются пользователем.
4. `processing` выполняет финальную категоризацию и сохранение.

### Сценарий C: Контроль бюджета и целей
1. `analysis-control` читает агрегированные финансовые данные.
2. Формирует показатели, отчёты и статус целей.
3. При отклонениях инициирует уведомления.

## 7) Нефункциональные требования

- **Безопасность:** JWT + refresh, TLS 1.3, аудит операций.
- **Наблюдаемость:** Sentry, Prometheus/Grafana, structured logging.
- **Синхронизация:** Pull-to-Refresh, фоновые синки, оффлайн-очередь, Redis-кэш.
- **Производительность:** выделение ML в отдельный сервис для независимого масштабирования.

## 8) Ограничения текущего этапа

- Нет прямой Open Banking интеграции на этапе 1.
- Подписки определяются косвенно по транзакционным паттернам.
- Качество категоризации ориентировано на целевой уровень ~94% на синтетических данных.

## 9) Рекомендации по эволюции (без смены принятых решений)

- Утвердить контракт версионирования между Go-сервисами и `ml-service` (schema version в payload).
- Ввести централизованный event-log для межсервисной трассировки доменных событий.
- Добавить quality-gates для ML (drift, confidence thresholds, golden set regression).
- Укрепить стратегию миграций БД: backward-compatible миграции + phased rollout.
